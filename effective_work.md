# RUST EFFECTIVE WORK POLICY (ThinkPadBar)

Этот документ определяет протокол работы для ИИ-агентов и разработчиков в данном проекте.
Цель: максимальный прогресс при минимальном выводе (token efficiency) и быстрых проверках.

## 1. Fast Feedback Loop (Цикл проверки)

- **`cargo check` всегда перед `cargo run`**: Используй `cargo check` для проверки типов и синтаксиса. Это в 5-10 раз быстрее полной сборки и генерирует меньше шума.
- **Точечная компиляция**: Если проект станет многомодульным, используй `cargo check -p <package_name>`.
- **Тихая сборка**: Используй флаг `--quiet` (`-q`), чтобы скрыть прогресс-бары, оставляя только ошибки и предупреждения.

## 2. Поиск и Навигация (Search Hierarchy)

Не читай файлы целиком. Используй:
1. `rg --type rust "fn name"` — быстрый поиск определений функций.
2. `fd . src/` — обнаружение новых файлов модулей.
3. `sg` (ast-grep) — для поиска структурных паттернов (например, всех имплементаций `Message`).
4. `bat -r 10:50 src/main.rs` — чтение только нужного диапазона строк.

## 3. Управление выводом и логами

- **Cargo Errors**: При большом количестве ошибок читай только первые 1-2 ошибки (`cargo check 2>&1 | head -n 20`). Не дампи весь лог ошибок в контекст.
- **Runtime Logs**: Используй `RUST_LOG=warn` или `RUST_LOG=thinkpadbar=debug` для фильтрации вывода. Не запускай приложение с дефолтным шумным логированием.
- **JSON Data**: Для анализа `Cargo.lock` или метаданных используй `dasel` или `gron`.

## 4. Тестирование и Верификация

- **Targeted Tests**: Запускай только конкретный тест: `cargo test test_name`.
- **No Capture**: Если нужен отладочный вывод в тестах, используй `cargo test -- --nocapture`.
- **Benchmarking**: Не запускай `cargo bench` без явной необходимости, это долго и ресурсозатратно.

## 5. Работа с зависимостями

- **`cargo tree`**: Перед добавлением нового крейта проверь дерево зависимостей (`cargo tree -d`), чтобы избежать дублирования версий (как произошло с `iced_runtime`).
- **`cargo add`**: Используй `cargo add <crate> --features <feat>`, чтобы не редактировать `Cargo.toml` вручную.

## 6. Принцип "Surgical Edit"

- Делай минимально возможные изменения.
- Не исправляй `clippy` ворнинги в файлах, которые не относятся к текущей задаче.
- Используй `cargo fmt` только для измененных файлов.

---
**Правило золотого сечения:** Один `cargo check` экономит 1000 токенов на перечитывании логов ошибок сборки.
